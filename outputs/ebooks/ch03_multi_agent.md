# 第三章：多代理系統——讓 AI 自己組團隊

## 一個人的極限

上一章你學會把一個 Agent 當員工用。但現實中，你不是只有一個員工。

你有：
- 找資料的
- 寫內容的
- 校稿的
- 做圖的
- 排版的

如果這些人要「一個做完換下一個」，那你還是很慢。

多代理系統的核心就一句話：**全部同時動。**

---

## 什麼是 Multi-Agent？

> 不是一個 AI 做完所有事。是五個 AI 各自做各自的，做完後自動整合。

就像一間公司：
- 研究部：搜尋市場資料
- 內容部：根據資料寫草稿
- 編輯部：檢查錯字、統一口氣
- 設計部：生出封面規範和排版規格
- 品管部：檢查所有產出，不合格退回重做

你不干涉過程。你只驗收結果。

---

## 怎麼拆任務？關鍵是「互不依賴」

一個爛的拆分：

```
研究 → 等研究做完 → 寫作 → 等寫作做完 → 校稿 → 等校稿做完 → 排版
```

一個卡住，全軍覆沒。

一個好的拆分：

```
        ┌→ Agent A：搜尋市場資料
目標 ──→ Agent B：搜尋寫作素材  ⟩ 同時開跑
        └→ Agent C：搜尋上架規範

（三者完成後）

        ┌→ Agent D：寫第一章
結果 ──→ Agent E：寫第二章  ⟩ 同時開跑
        └→ Agent F：寫第三章

（全部完成後）

最終 → Agent G：統整、校稿、存檔
```

**規則：同一層的不互相等。** 等的那層越少越好。

---

## 實際案例：出一本童書

不用 Agent 的做法（一個人做）：

1. 想故事 → 2 小時
2. 寫文案 → 1 小時
3. 做封面規範 → 1 小時
4. 校稿 → 30 分鐘
5. 上架 → 30 分鐘
**總共 5 小時，你全程在電腦前。**

用多代理的做法（你當老闆）：

1. 輸入：「出一本 3-6 歲童書，主題是收玩具的好習慣，雙貓主角，繁體中文，上架 Readmoo」
2. 三個 Agent 同時開工：
   - Agent A：搜尋市場上同類童書的定價和風格
   - Agent B：根據主題寫故事草稿
   - Agent C：產出封面設計規範
3. 完成後，兩個 Agent 接手：
   - Agent D：校稿、優化文字
   - Agent E：排版成上架格式
4. Agent F：檢查全部，上架

**你花的時間：2 分鐘（輸入任務）。**

---

## 什麼任務適合多代理？什麼不適合？

| 適合 | 不適合 |
|------|--------|
| 內容創作（書、文章、文案） | 單一簡單問題（「今天天氣？」） |
| 研究報告 | 即時對話 |
| 網站製作 | 需要人類品味判斷的事 |
| 大量資料處理 | 一次性的小任務 |

簡單記：**如果一件事要花你超過一小時，就值得拆給多代理。**

---

## 工具選擇：你不是只能自己寫 code

現在市面上已經有現成的多代理框架（開源的），你不用從零造輪子。但你要知道原理——因為框架會變，架構思維不會。

這本書不推特定工具。你學會的是「怎麼拆任務、怎麼讓代理不互卡」，這比任何工具都值錢。

---

## 這一章的重點

- 多代理 = 同時開工 + 互不依賴
- 拆任務的原則：同層並行、跨層串行
- 你的角色從「做事的人」變成「設目標 + 驗收的人」
- 適合多代理的任務：內容創作、研究、大量處理
